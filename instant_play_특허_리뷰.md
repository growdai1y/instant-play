# Instant Play 직무발명서 신규 특허 리뷰

> 검토 대상: `instant_play_직무발명서.md` 현재본<br>
> 검토 기준일: 2026-07-21<br>
> 선행기술 판단 기준일: 미확정 — 발명의 우선일·최초 공개일·예정 출원일 확인 필요<br>
> 검토 성격: 신규성·진보성·기재 충실도·청구항 구조·예비 권리충돌 위험 검토<br>
> 결론: 기존 리뷰는 전부 폐기하며, 이 문서가 이를 대체한다.

## 1. 검토 원칙과 결론

이 리뷰는 직무발명서 현재본의 기재와 청구항을 처음부터 다시 읽고, 외부 VOD 콘텐츠 통합, 보호된 스트리밍 접근, DRM, 단말 호환성 선택, 콘텐츠 재생 검증과 직접 관련된 특허만을 기준으로 작성했다. 기존 리뷰의 발명 정의, 용어, 선행기술 평가는 판단 근거로 사용하지 않았다.

발명의 특허법상 기준일이 아직 제공되지 않았으므로 아래 문헌은 우선 **선행기술 후보 문헌**으로 분류한다. 각 패밀리의 공개일이 발명의 기준일보다 앞서는지 확인한 뒤에야 확정 선행기술로 사용할 수 있다.

표시의 의미는 다음과 같다.

- `[기재]`: 직무발명서에 실제로 적혀 있는 내용
- `[선행]`: 선행특허에서 확인되는 내용
- `[평가]`: 위 자료를 기초로 한 특허성 또는 권리위험 판단
- `[확인필요]`: 실제 제품·계약·API 동작을 확인해야 결론을 낼 수 있는 사항
- `[제안]`: 현재 기재 범위 안에서의 명세서·청구항 정리 방향

### 종합 판단

| 항목 | 판단 | 이유 |
|---|---|---|
| 발명의 실체 | 파악 가능 | 업체 단위 연동정보, 콘텐츠 단위 등록정보, 사전 재생 검증, 추천 활성화, 실행 시점 권한 취득과 재생정보 전달이 하나의 수명주기로 연결되어 있다. |
| 신규성 | 잠정적 긍정 여지, 기준일 확인 필요 | 조사된 단일 문헌에서 현재 청구항 1의 독립항 구성 전체가 그대로 확인되지는 않았다. 사전 시험재생과 그 결과에 따른 추천 등록은 청구항 1이 아니라 청구항 8·17의 추가 구성으로 별도 평가했다. |
| 진보성 | 쟁점 큼, 구체 결합분석 전 판단 유보 | 복수 제공자 콘텐츠 통합, 외부 제공자 접근, 권한·DRM 처리, 단말 호환성 선택, 재생 전 검증은 각각 알려져 있어 복수문헌 결합 거절 시나리오를 대비해야 한다. |
| 현재 청구항 품질 | 전면 재정리 필요 | 청구항 1이 지나치게 길고, 방법 단계가 아닌 데이터 정의가 단계로 기재되어 있으며, 다수의 부가구성이 독립항에 누적돼 있다. |
| 명세서 정합성 | 보완 필요 | 등록·검증 시점, 요청·응답 상관관계, 필수 보안 필드, 유효시간, 단말 결합정보 등에서 서로 맞지 않는 부분이 있다. |
| 최신 피드백 | 방향 타당, 구조화 필요 | 콘텐츠별 사전조건은 이미 있으나, 조건 충족으로 생성되는 업체 제공용 결과 데이터와 업체의 처리행위가 조건별로 결합되어 있지 않다. 또한 등록 시 검증된 재생구성과 실행 시 재생구성의 관계를 현재 문언보다 강하게 제한해야 한다. |
| 예비 권리충돌 위험 | 판단 유보, 중요 확인대상 존재 | 사용자 경험이 가까운 활성 미국 등록특허와 권한·DRM·단말선택의 각 하위 구조에 관한 활성 권리가 존재한다. 출시국과 제품 구조를 확정한 뒤 등록 권리 대조가 필요하다. |
| 출원 판단 | 출원 검토 가치는 있으나 현재는 판단 유보 | 기준일과 발명자가 인식한 구체 구조를 확인하고 청구항을 다시 분석한 뒤 출원 여부를 결정해야 한다. 현 상태 제출은 권하지 않는다. |

핵심 결론은 단순하다.

1. “여러 VOD를 한 화면에서 추천하고 곧바로 재생한다”는 사용자 경험만으로는 넓은 권리를 얻기 어렵다.
2. 광고·결제·프로모션 같은 시청 전 조건, 짧은 접근 토큰, DRM 라이선스, 코덱·DRM 호환성 선택도 각각 알려진 기술이다.
3. 최신 피드백은 타당하다. 현재 문서에는 콘텐츠별 사전조건, 조건 충족 정보 및 정산 이벤트가 각각 있으나, **어떤 조건의 충족이 어떤 결과를 만들고, 그 결과 중 무엇을 외부 VOD 제공자에 전달하며, 제공자가 무엇을 확인·기록한 뒤 어떤 권한을 발급하는지**가 하나의 데이터관계로 정의되어 있지 않다.
4. 조건 충족의 최종 판정은 플랫폼이 담당하고, 외부 VOD 제공자는 플랫폼이 보낸 조건 결과 데이터가 등록된 조건 결과 규칙과 업체 고유 권한정책에 맞는지를 확인한 후 원 콘텐츠의 재생 권한을 발급하는 것으로 역할을 구분해야 한다.
5. 실행 시점에 외부 VOD 제공자가 등록 단계에서 검증되지 않은 새로운 재생방식을 자유롭게 선택하는 현재 문언은 최신 피드백과 충돌한다. 콘텐츠별로 등록 단계에서 실제 통과한 재생구성과 프로필 버전을 저장하고, 실행 시점에는 그 검증된 구성집합과 단말 지원범위에 속하는 구성만 사용하도록 바꾸는 것이 타당하다.
6. 광고 정산비율이나 프로모션 비용부담 자체는 사업규칙이므로 그것만으로 특허성이 강해지지는 않는다. 보호 중심은 **조건 프로파일에 결합된 결과 데이터 규격, 플랫폼의 판정에 따른 결과 데이터 생성, 제공자 측 확인·권한 발급, 등록 시 검증된 재생구성과 실행 시 응답의 일관성 제어**라는 서버 간 처리관계에 두어야 한다.
7. 실제 제품 완성은 출원의 요건이 아니지만, 이 연결관계를 발명자가 구체적으로 인식하고 문서가 통상의 기술자가 실시할 수 있을 정도로 뒷받침해야 한다.

## 2. 직무발명서에서 확인되는 실제 발명

### 2.1 전체 흐름

`[기재]` 현재 직무발명서는 다음 흐름을 제시한다.

1. 외부 VOD 제공자가 업체 식별자, 권한 요청 규격, 지원 스트리밍 방식, 코덱 및 DRM 연동정보를 포함하는 업체 단위 정적 재생 연동 프로필을 등록한다.
2. 개별 콘텐츠는 업체 콘텐츠 식별자, 플랫폼 콘텐츠 식별자, 추천용 메타데이터, 조건 프로파일 및 업체 연동 프로필 참조값과 함께 별도로 등록된다.
3. 카탈로그용 등록정보에는 실행 시점의 실제 콘텐츠 URL·매니페스트, 콘텐츠 접근 토큰, DRM 키 또는 해당 재생을 위한 DRM 요청정보를 두지 않는다고 설명한다.
4. 플랫폼은 등록된 업체 콘텐츠 식별자와 연동 프로필을 이용하여 검증용 권한을 받고 실제 재생을 시험한다.
5. 검증에 성공한 콘텐츠만 추천 후보가 되고, 연동 프로필 또는 관련 버전이 바뀌면 검증 결과를 무효화하고 추천 상태를 다시 판단할 수 있다고 설명한다.
6. 이용자가 콘텐츠를 선택하면 플랫폼이 해당 콘텐츠의 적용 조건을 정하고, 조건 실행 결과가 충족됐는지 확인한다.
7. 그 후 플랫폼이 외부 VOD 제공자에 업체 콘텐츠 식별자, 직무발명서가 `조건 충족 증빙 정보`라고 정의한 데이터, 연동 프로필 참조값과 단말 재생 능력정보를 보내 한시적 재생 권한을 요청한다.
8. 외부 VOD 제공자는 등록 프로필과 단말 재생 능력의 공통 지원범위 안에서 스트리밍·DRM·코덱 구성을 선택하고, 한시적 권한과 실제 재생정보를 반환한다.
9. 플랫폼은 반환된 구성이 등록 프로필과 단말 능력 범위 안인지 확인한 뒤 이용자 단말의 재생기에 전달한다.
10. DRM 콘텐츠의 키는 플랫폼에 평문으로 전달되지 않고, 재생기가 제공자와 연계된 DRM 시스템에서 라이선스를 취득한다고 설명한다.

### 2.2 발명의 구성요소를 중요도별로 분리한 결과

`[평가]` 현재 문서에는 서로 다른 가치의 구성이 섞여 있다.

#### A. 특허성 판단의 중심으로 삼을 수 있는 구성

- 업체 단위 연동 프로필과 콘텐츠 단위 등록정보의 역할 분리
- 사전 권한 취득 및 실제 재생 성공 여부를 추천 활성 상태와 연결하는 구성
- 등록 시 실제 통과한 콘텐츠별 재생구성과 프로필 버전을 실행 시점 권한 응답의 허용기준으로 다시 사용하는 구성
- 콘텐츠별 조건 프로파일을 조건 결과 데이터 규격 및 외부 VOD 제공자의 확인·권한 발급행위와 연결하는 구성
- 실제 재생정보를 콘텐츠 등록정보와 분리하여 이용자 선택 이후에 취득하는 시간적 구조

`[평가]` 위 세 번째·네 번째 구성은 최신 피드백에 따른 보강이 필요하다. 현재 문서의 “넓은 등록범위 안에서 업체가 실행 시점에 새 재생방식을 선택”하는 문언과 불투명한 `proofPayload`만으로는 위 중심구성이 완성되지 않는다.

#### B. 중요하지만 단독 차별성이 약한 구성

- 광고, 결제, 프로모션, 쿠폰 또는 멤버십 확인을 재생 전 조건으로 사용하는 것
- 짧은 유효시간의 콘텐츠 URL 또는 접근 토큰
- DRM 라이선스 취득과 콘텐츠 키의 비노출
- 단말의 스트리밍·DRM·코덱·해상도 지원정보 확인
- 여러 외부 VOD 콘텐츠를 한 플랫폼에서 추천하는 것
- 제공자 전용 앱 설치나 별도 자격정보 입력을 줄이는 사용자 경험

#### C. 권리범위를 오히려 약하게 만들 수 있는 구성

- 특정 저장소에 특정 정보가 없다는 부정형 한정
- 광고·결제·정산 필드의 장황한 열거
- 구현마다 쉽게 바뀌는 JSON 필드명, 내부 식별자 및 유효시간 수치
- 플랫폼, 외부 VOD 제공자, 단말 및 DRM 시스템의 행위를 한 청구항에 과도하게 누적하는 것

`[기재]` 청구항 구조는 구분해서 읽어야 한다. 청구항 1의 필수 흐름은 적용 조건 프로파일 선택, 조건 실행 결과 확인, 그 결과에 결부된 `조건 충족 증빙 정보` 생성, 그 이후의 외부 VOD 제공자 권한 요청을 포함한다. 반면 검증용 권한에 의한 실제 재생시험과 시험 성공 콘텐츠만 추천 시스템에 등록하는 구성은 청구항 8 및 청구항 17에서 추가된다.

### 2.3 최신 피드백의 타당성

| 피드백 | 판단 | 검토 결과 |
|---|---|---|
| 플랫폼이 사전조건 충족을 주도적으로 확인 | 타당 | 현재 설명과 청구항 1에는 플랫폼의 확인이 있으나, 단말의 조건 실행 확인부와 청구항 18도 최종 판정자처럼 읽힌다. 단말은 원천 실행 결과를 수집·보고하고, 플랫폼이 콘텐츠별 조건 프로파일에 따라 충족 여부를 최종 판정하는 것으로 통일해야 한다. |
| 조건 충족 시 생성되는 결과와 업체 제공 데이터를 조건별로 명시 | 매우 타당 | 현재 조건 프로파일에는 조건 유형과 정산 참조가 있고 별도의 조건 충족 레코드와 정산 이벤트도 있으나, 업체 요청의 `proofPayload`와 `settlementEventId`만으로는 광고 정산, 업체 부담 무료 프로모션, 플랫폼 부담 프로모션의 서로 다른 결과를 업체가 해석하기 어렵다. |
| 업체가 결과 데이터의 유효성을 판단한 뒤 권한 발급 | 타당하되 역할 구분 필요 | 플랫폼은 `사전조건 충족 여부`를 판정하고, 업체는 수신 데이터가 등록된 결과 규격·정산 규칙·대상 콘텐츠 및 업체 고유 권한정책에 맞는지를 확인해야 한다. 업체가 동일 조건을 처음부터 다시 판정하는 구조로 쓰면 주체가 다시 충돌한다. |
| 업체가 결과 데이터를 보관해 사후 정산 가능하게 함 | 타당, 종속 권리축 권고 | 조건 판정·권한 발급·실제 재생·정산 근거가 동일 거래로 연결되면 실시형태로 의미가 있다. 다만 금액·분배율 계산만을 독립항 중심에 두면 사업방법으로 평가될 위험이 크므로, 식별자와 상태를 연결하는 서버 간 기록처리를 중심으로 써야 한다. |
| 등록 시 확인한 재생방식과 실행 시 방식이 달라지면 등록정보 불일치로 처리 | 타당 | 현재 문서는 등록 시 넓은 프로필을 시험한 뒤 업체가 실행 시점에 새 조합을 선택할 수 있게 쓰여 있어 피드백과 충돌한다. 실제 통과한 콘텐츠별 재생구성 및 프로필 버전을 저장하고, 검증집합 밖 응답은 재생기로 전달하지 않는 구조가 필요하다. |
| 실행 시 차이는 업체의 실수 | 설명상 원인 예시는 가능하나 청구항 표현으로는 부적절 | 업데이트 누락 외에도 전파 지연, 잘못된 응답, 캐시 또는 설정 불일치가 원인일 수 있다. 권리문언은 업체 과실을 단정하지 말고 `등록 버전 불일치` 또는 `검증되지 않은 실행 구성`이라는 관측 가능한 상태와 차단·재검증 동작을 규정해야 한다. |

`[평가]` 따라서 피드백은 발명의 방향을 바꾸는 임의의 추가가 아니라, 현재 문서에 흩어져 있는 조건 프로파일, 조건 충족 정보, 정산 이벤트, 업체 권한 발급 및 등록 시 재생검증을 서로 결합하는 정합성 보완이다. 다만 아래 진화안은 아직 직무발명서 현재 청구항의 확정 구성으로 보아서는 안 되며, 발명자가 이 구조를 구체적으로 인식한 시점과 실제 설계를 확인한 뒤 반영해야 한다.

### 2.4 권고하는 발명 진화 구조

#### 2.4.1 조건 판정과 업체 권한 판단의 경계

| 주체 | 최종 책임 | 입력 | 출력 |
|---|---|---|---|
| 단말·광고·결제·프로모션 시스템 | 조건 실행 및 원천 결과 보고 | 조건 실행 정보 | 광고 완료, 결제 승인, 캠페인 적용 등 원천 실행 결과 |
| 플랫폼 | 선택된 콘텐츠의 사전조건 충족 여부 최종 판정 | 콘텐츠별 조건 프로파일과 원천 실행 결과 | 조건 충족 판정 및 조건별 결과 데이터 |
| 외부 VOD 제공자 | 수신 결과 데이터와 업체 고유 권한정책의 적합성 판단 | 조건별 결과 데이터, 권한 요청, 등록된 결과 규칙 | 승인 또는 거절, 한시적 재생 권한 및 실행 시점 재생정보 |
| 정산 시스템 | 권한·재생 결과에 따른 정산상태 반영 | 조건 결과 식별자, 업체 권한 식별자, 재생 결과 | 확정·취소·조정된 정산 기록 또는 정산 없음 상태 |

`[제안]` 플랫폼의 판정은 광고 시청·결제·프로모션이라는 **플랫폼이 관찰하는 사전조건의 충족**에 관한 최종 판정이다. 콘텐츠 제공기간, 지역권리, 동시시청, 계약 상태와 같은 외부 VOD 제공자의 고유 권한정책까지 플랫폼이 대신 결정한다는 의미가 아니다. 외부 VOD 제공자는 플랫폼 판정과 조건 결과 데이터를 입력으로 사용하되, 자신의 권한정책을 통과한 경우에만 원 콘텐츠 권한을 발급한다.

#### 2.4.2 콘텐츠별 조건과 결과를 함께 등록하는 구조

`[기재]` 현재 `entitlementConditionProfiles`는 콘텐츠별 조건 유형, 실행 파라미터, 권한 범위, 정산 참조 및 권한 요청 대상을 포함한다.

`[제안]` 각 조건 프로파일에 다음 성격의 **조건 결과 프로파일**을 결합해야 한다. 새 객체로 분리하거나 기존 조건 프로파일 내부 필드로 둘 수 있으나, 어느 방식을 택하든 조건과 결과 사이의 참조관계가 하나로 고정되어야 한다.

| 항목 | 의미 |
|---|---|
| 결과 유형 | 광고 정산 결과, 업체 부담 무료 프로모션 결과, 플랫폼 부담 프로모션 결과, 사용자 결제 결과 등 |
| 결과 데이터 규격 | 조건이 충족됐을 때 업체에 제공할 공통 필드와 조건별 필드 |
| 업체 확인 규칙 | 결과 데이터가 어떤 콘텐츠·조건 프로파일·캠페인·정산 규칙과 일치해야 하는지 |
| 업체 처리행위 | 조건 결과 수락 시 권한 발급, 정산 근거 기록, 정산 없음 기록 등 |
| 비용부담 및 정산 처리 | 플랫폼 부담, 업체 부담, 광고수익 배분, 사용자 부담 또는 별도 정산 없음 |
| 적용 버전 | 조건 규칙과 결과 규격이 변경됐을 때 어느 버전을 적용할지 |

`[제안]` 실행 시에는 불투명한 `proofPayload`만 전송하지 말고, 현재 조건 충족 레코드의 공통정보와 조건별 경제적·기술적 결과를 결합한 **조건 결과 데이터**를 생성해야 한다. 최소한 콘텐츠, 업체, 세션, 조건 프로파일과 버전, 판정 결과, 완료시각, 결과 유형, 적용 정산 규칙 또는 정산 없음 표시, 업체가 확인할 조건별 결과값 및 플랫폼 판정값을 포함하는 구조가 적절하다. 구체 필드명은 구현에 맞춰 정해야 하며, 모든 필드를 독립항에 열거할 필요는 없다.

#### 2.4.3 조건 유형별 결과 데이터

| 사전조건 | 플랫폼의 충족 판정 | 업체에 제공할 결과 데이터의 예 | 업체의 후속 처리 |
|---|---|---|---|
| 광고 시청 | 지정 광고가 요구된 기준까지 재생됐는지 확인 | 광고·캠페인 식별정보, 완료시각, 적용 정산 규칙, 플랫폼·업체 배분 기준 또는 계산 결과, 정산 이벤트 참조 | 데이터와 등록 규칙을 확인하고 정산 근거를 기록한 뒤 권한 발급 |
| 업체 부담 무료 프로모션 | 해당 업체 캠페인과 콘텐츠가 적용 대상인지 확인 | 업체 캠페인, 콘텐츠, 적용 기간·대상 및 플랫폼 비용부담 없음 표시 | 업체 프로모션 규칙과 일치하면 별도 콘텐츠 비용 정산 없이 권한 발급 |
| 플랫폼 부담 프로모션 | 플랫폼 캠페인의 예산·적용조건이 충족됐는지 확인 | 플랫폼 캠페인, 비용부담 주체, 업체에 지급할 콘텐츠 대가 또는 계산 규칙, 후정산 근거 식별정보 | 업체가 플랫폼 부담 근거를 기록하고 권한 발급 후 정산대상으로 관리 |
| 사용자 결제 | 결제 승인과 대상 콘텐츠·금액의 일치 확인 | 결제 승인 정보, 금액·통화, 적용 수수료 또는 배분 규칙, 정산 참조 | 승인 데이터와 업체 규칙을 확인하고 권한 발급 및 정산 처리 |

`[평가]` 광고 식별정보, 배분비율, 콘텐츠 비용 같은 값은 그 자체로 기술적 차별점이 강하지 않다. 특허상 중심은 콘텐츠별 조건 결과 프로파일에 따라 플랫폼이 서로 다른 형식의 결과 데이터를 생성하고, 외부 VOD 제공자가 그 데이터를 등록 규칙과 결부해 권한 발급과 정산상태 처리의 입력으로 사용한다는 시스템 관계다.

#### 2.4.4 등록 시 재생검증과 실행 시점 처리

`[기재]` 현재 문서는 콘텐츠가 하나 이상의 정적 재생 연동 프로필을 참조하고, 등록 뒤 실제 재생을 시험하며, 실행 시점에는 외부 VOD 제공자가 프로필과 단말 능력의 공통범위에서 스트리밍·DRM·코덱 조합을 선택한다고 쓴다. 검증 결과에는 주로 검증된 프로필 식별자가 남아, 하나의 넓은 프로필 중 실제로 어떤 조합이 통과했는지는 불명확하다.

`[제안]` 다음과 같이 바꾸는 것이 피드백과 실제 기술위험에 맞다.

1. 등록 검증 결과에 업체·콘텐츠·정적 프로필 식별자와 버전뿐 아니라 실제 시험에 통과한 스트리밍·DRM·코덱 등의 **검증된 재생구성집합**을 연결한다.
2. 한 콘텐츠에 하나의 재생구성만 등록·검증됐다면 실행 시점에 재생방식을 새로 선택하는 단계는 두지 않는다.
3. 복수 구성이 검증됐다면 실제 구현 주체가 단말 지원범위에 속하는 하나를 고르되, 그 선택은 반드시 `검증된 재생구성집합` 안에서만 이루어져야 한다. 실제 구현이 플랫폼 선택이라면 플랫폼이 구성 식별자를 권한 요청에 포함하고, 업체는 그 구성에 대한 권한과 동적 재생정보를 반환하도록 쓰는 것이 가장 정확하다.
4. 업체 응답의 프로필 버전 또는 실행 구성이 검증집합 밖이면 플랫폼은 동적 재생정보를 재생기에 전달하지 않고 콘텐츠 상태를 업데이트 필요 또는 재검증 필요 상태로 전환한다.
5. 업체가 지원방식을 바꾸려면 콘텐츠 또는 프로필 등록정보를 먼저 갱신하고 다시 검증받도록 한다.

`[평가]` 실행 시 매번 달라지는 매니페스트 위치, 접근 토큰, 권한 식별자 또는 DRM 요청정보의 값 자체를 등록값과 같게 만들 필요는 없다. 비교 대상은 정적으로 식별·버전 관리할 수 있는 재생구성이고, 동적 값은 그 검증된 구성에 따라 해당 세션용으로 발급되는 정보다. 이를 구분하지 않으면 정상적인 동적 URL·토큰 변경까지 업체 등록오류로 오판할 수 있다.

## 3. 특허성 판단 기준

`[평가]` 신규성은 원칙적으로 하나의 선행문헌이 청구항의 모든 구성을 직접적·명확하게 보여주는지로 판단하고, 진보성은 통상의 기술자가 복수 문헌과 통상의 지식을 이용해 쉽게 결합할 수 있는지로 판단해야 한다. 대한민국 출원은 [특허법 제29조 관련 공식 해설](https://www.law.go.kr/LSW/cgmExpcInfoP.do?cgmExpcDatSeq=6506624&mode=2&ofiClsCd=350137)과 [특허청 심사기준](https://kipo.go.kr/ko/kpoContentView.do?menuCd=SCD0201119)에 맞춰 최종 청구항을 평가해야 한다.

광고료 배분, 프로모션 비용부담 및 정산 약정은 사업규칙의 성격이 강하다. [지식재산처의 영업방법 발명 안내](https://www.kipo.go.kr/ko/kpoContentView.do?menuCd=SCD0200230)와 [컴퓨터관련 발명 안내](https://www.kipo.go.kr/ko/kpoContentView.do?menuCd=SCD0200238)를 고려하면, 출원안은 사업 합의 자체가 아니라 플랫폼 서버·외부 VOD 제공자 시스템·저장소·인터페이스가 어떤 데이터를 입력받아 어떤 판정·상태전이·출력을 만드는지로 구체화해야 한다.

이 리뷰의 조사는 출원 전 예비조사다. 조사된 문헌에서 전체 구성이 발견되지 않았다는 사실은 절대적인 신규성 보증이 아니다. 한국어·영어·중국어 문헌, 특허 패밀리, 비특허문헌 및 비공개 선출원까지 포함한 정식 조사가 별도로 필요하다.

## 6. 신규성 및 진보성 판단

### 6.1 단일문헌 기준 신규성

#### 현재 청구항 1

`[평가]` 청구항 1의 핵심 연결관계를 요약하면 다음과 같다. 이 요약 외에도 청구항 1에는 조건 프로파일의 세부 정의 등 추가 한정이 있다.

1. 업체 단위 정적 연동 프로필과 콘텐츠 단위 등록정보의 분리
2. 콘텐츠 등록정보 또는 상시 카탈로그에 실제 본편 위치, DRM 키 및 해당 재생을 위한 DRM 요청정보를 두지 않는다는 한정
3. 콘텐츠 메타데이터와 플랫폼 콘텐츠 식별자를 추천 시스템에 제공하고 이용자 선택을 수신
4. 선택 콘텐츠의 적용 조건을 정하고 조건 실행 결과의 충족을 확인
5. 해당 결과에 결부된 `조건 충족 증빙 정보`가 생성된 경우에만 업체 콘텐츠 식별자·연동 프로필·단말 재생 능력정보를 이용해 외부 VOD 제공자에 재생 권한 요청
6. 외부 제공자가 등록 프로필과 단말 지원범위의 공통범위에서 재생방식을 선택하고 한시적 권한과 실행 시점 재생정보를 반환
7. 플랫폼이 반환된 재생방식을 등록 프로필 및 단말 지원범위와 다시 대조한 뒤 재생기에 전달

예비 조사에서는 청구항 1의 모든 한정을 하나로 공개한 문헌을 확인하지 못했다. 다만 발명의 기준일이 미확정이고 조사가 완결된 것도 아니므로 신규성 판단은 유보한다. 또한 청구항 1은 많은 단계를 누적해 범위가 좁다. 구성 하나를 빼는 경쟁 구현에 쉽게 회피될 수 있고, 경쟁 서비스가 모든 단계를 수행하지 않으면 권리행사도 어렵다.

#### 청구항 8의 추가 구성

`[평가]` 청구항 8은 청구항 1에 외부 제공자의 검증용 권한을 이용한 실제 재생시험과 시험 성공 콘텐츠의 추천 등록을 추가하는 종속 방법항이다. US9081939B2의 명세서는 권리 확인과 DRM 재생을 거친 QA 제공상태에서 운영 제공상태로 전환하는 흐름을 설명하고, CN115002449B는 플랫폼 주입 뒤 지역별 온라인 재생 문자열·링크·재생데이터를 검사한다. 따라서 “시험재생 후 추천 등록”만 떼어 넓게 주장하기는 어렵다.

청구항 8은 청구항 1의 외부 제공자 등록 프로필, 조건 확인 후 권한 요청, 제공자 측 구성선택 및 플랫폼 확인에 시험재생·추천 등록을 더한다. 예비 조사상 단일문헌 대응은 더 어려워질 수 있지만, 필수 단계가 늘어나 권리범위도 더 좁아진다.

#### 청구항 17의 추가 구성

`[평가]` 청구항 17은 청구항 16의 시스템에 검증용 권한 취득과 프로필별 시험재생을 수행하는 검증부 및 성공 콘텐츠의 추천 등록을 추가한다. 방법항인 청구항 1 또는 8과 직접 결합된 항이 아니므로, 청구항 16의 시스템 구성 전체를 기준으로 별도 신규성·진보성 대조가 필요하다.

### 6.2 복수문헌 결합 기준 진보성

`[평가]` 아래는 확정 결론이 아니라 심사관이 제시할 수 있는 두 가지 거절 시나리오다.

#### 시나리오 A — 외부 VOD 접근에서 출발

- **출발 문헌**: EP2850841B1 또는 US12101309B2는 외부 제공자 콘텐츠를 통합 목록에서 선택하고 외부 콘텐츠 접근을 연결하는 기본구조를 제시한다.
- **출발 문헌에 없는 부분**: 현재 청구항 1의 조건 실행 결과 확인과 `조건 충족 증빙 정보` 생성, 사전 등록 프로필과 단말 능력의 공통범위에서 이루어지는 외부 제공자 측 구성선택, 플랫폼의 반환값 확인이 한 흐름으로 나타나지 않는다.
- **결합 후보**: US7711647B2 또는 US9450934B2의 사업규칙·권한·DRM 연결과, US12111891B2 또는 US9129092B1의 단말 호환 재생구성 선택을 결합할 수 있다는 논리가 예상된다.
- **객관적 기술과제와 결합 이유**: 보호된 외부 VOD를 서로 다른 DRM·코덱 능력의 단말에 제공하면서 허용되지 않거나 재생 불가능한 구성을 전달하지 않는 과제는 결합 이유가 될 수 있다.
- **결합 방해요인과 반론**: EP2850841B1은 사전 등록 프로필이 아니라 기존 합의가 없는 콘텐츠에 대한 임시 합의를 중심으로 하고, US12101309B2는 제공자 계정 접근자격의 사전 보관을 전제로 한다. US9450934B2의 SSO 기반 다단계 신뢰구조도 현재 문서의 제공자별 연동 API와 다르다. 이들을 현재 청구항의 데이터관계와 주체분담으로 바꾸는 것이 단순 치환인지 논쟁할 여지가 있다.

#### 시나리오 B — 콘텐츠 QA와 제공상태 전환에서 출발

- **출발 문헌**: US9081939B2는 제공자 식별자에 따른 콘텐츠 처리, 권리 확인과 DRM 재생을 포함한 QA, 운영 제공상태의 활성화를 설명한다.
- **출발 문헌에 없는 부분**: 원본 미디어를 플랫폼이 수집하지 않는 외부 제공자 참조형 구조, 추천 가능 상태, 이용자 조건 확인 뒤 외부 제공자가 실행 시점 권한·재생정보를 반환하는 흐름이 그대로 나타나지 않는다.
- **결합 후보**: EP2850841B1 또는 US12101309B2의 외부 콘텐츠 통합과 US12111891B2의 단말 재생선택 기술을 결합할 수 있다는 논리가 예상된다.
- **객관적 기술과제와 결합 이유**: 품질검사를 통과한 외부 콘텐츠만 노출하고 이용자 단말에 호환되는 보호구성으로 전달하려는 과제는 결합 이유가 될 수 있다.
- **결합 방해요인과 반론**: US9081939B2는 플랫폼이 실제 미디어 자산과 CDN 위치를 관리하는 수집형 구조이고, 현재 문서는 원본 자산과 최종 권한통제를 외부 제공자에 남긴다. 자산 보유주체와 신뢰경계가 달라 결합 후에도 현재 문서의 전체 흐름이 바로 얻어지는지는 별도 판단이 필요하다.

두 시나리오 모두 성립 가능성은 있지만 자동으로 진보성 부정 결론이 되는 것은 아니다. 발명의 기준일을 확정한 뒤 실제 독립항 문언을 기준으로 결합 후에도 남는 구성과 예상 효과를 다시 대조해야 한다.

### 6.3 진보성을 방어할 수 있는 논리

`[제안]` 최신 피드백을 반영한 방어논리는 조건 유형의 나열이나 정산비율 자체가 아니라 다음 인과관계에 두어야 한다.

1. 콘텐츠별 조건 프로파일이 조건 판정 규칙뿐 아니라, 조건 충족 시 생성할 결과 유형·데이터 규격·비용부담·업체 확인 규칙·업체 처리행위를 함께 특정한다.
2. 이용자 선택 뒤 플랫폼은 단말이나 외부 시스템이 보고한 원천 실행 결과를 조건 프로파일과 대조하여 사전조건 충족 여부를 최종 판정한다.
3. 플랫폼은 선택된 조건의 결과 프로파일에 따라 광고 정산, 업체 부담 무료 프로모션, 플랫폼 부담 프로모션 또는 사용자 결제에 대응하는 조건 결과 데이터를 생성해 외부 VOD 제공자에 보낸다.
4. 외부 VOD 제공자는 그 데이터가 등록된 결과 규칙과 대상 콘텐츠에 맞는지 및 업체 고유 권한정책을 충족하는지 확인하고, 수락·정산상태와 재생 권한 발급을 연결한다.
5. 별도로 등록 단계에서는 콘텐츠별 실제 시험에 통과한 재생구성과 프로필 버전을 저장하고, 검증 성공 여부가 추천 가능 상태를 결정한다.
6. 실행 시점에는 검증된 재생구성집합과 단말 지원범위에 속하는 구성만 사용한다. 업체 응답이 해당 집합 또는 버전과 맞지 않으면 플랫폼은 재생정보 전달을 차단하고 콘텐츠를 업데이트 또는 재검증 대상 상태로 전환한다.
7. 동적 URL·접근 토큰·DRM 요청정보는 위 검증된 재생구성에 따라 세션별로 발급되며, 값이 동적이라는 이유만으로 등록 불일치로 처리하지 않는다.

이 구조가 구체적으로 기재된다면 단순히 광고 정산, 외부 권한 처리, 콘텐츠 QA 및 단말탐지를 나란히 붙인 것이 아니라 **콘텐츠별 조건의 충족이 타입화된 업체 제공 데이터와 외부 권한 발급을 만들고, 등록 시 검증된 기술상태가 추천 노출과 실행 시 재생허용을 함께 지배하는 교차 시스템 상태 제어**라고 설명할 수 있다.

`[평가]` 가장 큰 약점은 US7711647B2가 사업규칙 처리 후 외부 DRM 시스템의 데이터 확인·권리 발급과 사전 등록 프로필·단말정보 활용을 상당 부분 함께 보여준다는 점이다. 따라서 결과 유형이 네 가지라는 사실만으로는 부족하다. 조건과 결과 규격의 콘텐츠별 버전 결합, 업체 수락·정산상태·권한 발급의 연결, 등록 시 실제 통과 구성에 대한 실행 시점 불일치 차단이 모두 필수적인지 발명자의 설계에서 확인해야 한다.

`[확인필요]` 실제 구현에서 콘텐츠별 검증 재생구성을 하나만 저장하는지 복수로 저장하는지, 복수인 경우 플랫폼과 업체 중 누가 하나를 고르는지, 업체가 조건 결과 데이터의 어떤 값을 저장·확인하는지 및 프로필 변경 시 추천 상태를 해제하는지를 설계자료로 확인해야 한다. 제품 출시나 완성된 코드가 필수라는 뜻은 아니다. 발명자의 구체적 구상에 없는 연결은 위 진보성 논리에 사용하거나 직무발명서에 확정 구성으로 추가해서는 안 된다.

### 7.2 독립항 구조에 대한 권고

`[제안]` 최신 피드백은 현재 청구항의 단순 문구수정이 아니라 권리축 재배치를 요구한다. 아래 구조는 발명자 확인을 전제로 한다.

1. **플랫폼 방법 독립항**: 콘텐츠별 조건 프로파일과 결과 프로파일 저장 → 원천 실행 결과 수신 → 플랫폼의 조건 충족 판정 → 해당 결과 프로파일에 따른 조건 결과 데이터 생성·전송 → 업체 권한 응답 수신 → 응답의 재생구성이 콘텐츠별 검증 재생구성집합과 단말 지원범위에 속하는지 확인 → 일치 시 동적 재생정보 전달, 불일치 시 차단
2. **외부 VOD 제공자 방법 독립항**: 조건 결과 데이터와 권한 요청 수신 → 등록된 결과 규칙·대상 콘텐츠·업체 고유 권한정책에 따른 확인 → 수락·정산상태 기록 → 승인된 검증 재생구성에 대응하는 한시적 권한과 동적 재생정보 발급. 제공자 내부 행위는 침해 입증이 어려울 수 있으므로 플랫폼항보다 후순위다.
3. **등록·검증 수명주기 독립항 또는 병렬항**: 정적 재생 프로필·버전 등록 → 검증용 권한을 이용한 실제 재생시험 → 콘텐츠별 통과 재생구성집합 저장 → 성공 콘텐츠만 추천 가능 상태로 전환 → 프로필 변경 또는 실행 시 불일치에 따라 상태 무효화 → 갱신 등록 후 재검증
4. **플랫폼 시스템항**: 조건·결과 프로파일 저장부, 플랫폼 조건 판정부, 조건 결과 데이터 생성·전송부, 검증 재생구성 저장부, 업체 응답 확인 및 불일치 차단부로 플랫폼이 통제하는 구성만 포함
5. **조건별 종속항**: 광고 시청 결과와 광고 정산, 업체 부담 무료 프로모션과 별도 콘텐츠 비용 정산 없음, 플랫폼 부담 프로모션과 후정산 근거, 사용자 결제 승인과 정산을 각각 구체화
6. **정산 종속항**: 조건 결과, 업체 수락, 업체 권한 및 실제 재생 결과가 같은 거래로 연결되어 정산상태가 결정되는 처리
7. **재생구성 종속항**: 하나의 검증구성인 경우와 복수 검증구성인 경우를 나누고, 복수인 경우 실제 구현 주체가 검증집합과 단말 지원범위 안에서 하나를 선택하는 처리
8. **DRM 종속항**: 동적 재생정보의 지연 취득, 단말의 DRM 라이선스 취득 및 콘텐츠 키의 플랫폼 비노출
9. **데이터 등록 종속항**: 업체 레코드와 콘텐츠 레코드의 구분, 조건 프로파일과 결과 프로파일의 버전 참조, 검증 재생구성과 콘텐츠의 결합
10. **단말항**: 조건을 최종 판정하는 주체로 쓰지 말고, 조건 실행·원천 결과 보고 및 플랫폼이 허용한 동적 재생정보를 이용한 재생으로 한정

청구항을 짧게 만드는 것 자체가 목적은 아니다. 경쟁 시스템이 수행하는 핵심 연결은 포섭하되, 그 시스템이 수행하지 않아도 되는 부가 처리 때문에 전체 침해가 부정되지 않도록 계층화하는 것이 목적이다.

## 8. 명세서 내부 정합성 문제

아래 문제는 선행기술과 무관하게 출원 전에 바로 잡아야 한다.

### 8.1 등록 주체와 검증 시점

`[기재]` 콘텐츠 등록 JSON에 `verification.status=passed`가 포함되어 있으나, 다른 부분에서는 플랫폼의 검증 모듈이 등록 뒤 시험재생을 수행하고 그 결과를 만든다.

`[평가]` 제공자가 성공 상태를 등록하는 것인지, 플랫폼이 등록 후 생성하는 것인지 불명확하다. 추천 활성화의 법적·기술적 근거가 되는 상태이므로 생성 주체, 생성 시점, 갱신 주체를 통일해야 한다. 또한 `verifiedPlaybackProfileIds`만으로는 넓은 프로필 안에서 실제 어떤 스트리밍·DRM·코덱 조합이 시험에 통과했는지 알 수 없으므로, 콘텐츠·프로필 버전·실제 통과 재생구성을 검증 결과에 함께 연결해야 한다.

### 8.2 요청과 응답의 상관관계

`[기재]` 거절 기준은 콘텐츠, 세션, 조건 프로파일, 버전 및 검증값의 일치를 폭넓게 요구한다. 그러나 예시 권한 응답에는 `platformContentId`, 요청의 검증값, 업체·콘텐츠 등록 버전 등 일부 대조 필드가 없고 `requestId`는 포함되어 있다.

`[평가]` `requestId`로 서버에 보관된 원 요청을 조회해 대조하는 상태 저장형인지, 응답 자체의 반복 필드만으로 확인하는 상태 비저장형인지 구분해야 한다. 전자라면 요청상태의 매핑·보존기간·불일치 처리를, 후자라면 응답에 필요한 대조 필드를 설명해야 한다. 요청 필드, 응답 필드 및 검증 규칙을 하나의 표로 맞추는 것이 좋다.

### 8.3 무결성 필드의 선택사항 처리

`[기재]` `vendorSignature`, `platformSignature`, `proofSignature` 등은 일부 예시에서 선택 필드로 보이지만, 다른 부분의 신뢰성 설명은 서명 또는 무결성 검증을 사실상 전제로 한다.

`[평가]` 서명이 없는 허용 방식이 존재하는지, 대체 수단이 무엇인지, 어떤 응답을 거절하는지 구분해야 한다. 모두 선택사항으로 둔 채 강한 보안효과를 주장하면 기재가 약해진다.

### 8.4 유효시간의 관계

`[기재]` 예시에는 짧은 권한 유효시간과 그보다 긴 재생구간이 함께 나타난다.

`[평가]` 권한 요청을 시작할 수 있는 시간, 매니페스트 접근시간, DRM 라이선스 유효시간, 이미 시작한 구간을 재생할 수 있는 시간을 구분해야 한다. 그렇지 않으면 정상 실시가 중간에 실패할 수 있다.

### 8.5 단말 결합정보의 의미

`[기재]` `deviceBindingHash` 예시는 단말의 지원능력 목록에서 계산되는 것으로 읽힌다.

`[평가]` 동일 능력을 가진 여러 단말을 구분하려는 값인지, 단지 능력정보의 변조를 확인하려는 값인지 불분명하다. 명칭과 기능을 맞춰야 한다.

### 8.6 네트워크 위치정보의 범위

`[기재]` 콘텐츠 등록정보에는 대표 이미지가 포함될 수 있고, 동시에 실제 재생을 위한 네트워크 위치정보는 저장하지 않는다고 한다.

`[평가]` 대표 이미지 URL까지 금지하는 것으로 읽히지 않도록 “보호된 본편 스트림 또는 매니페스트에 접근하기 위한 실행 시점 위치정보”처럼 범위를 구분해야 한다.

### 8.7 중복 식별자와 충돌 규칙

`[기재]` `grantBinding.grantServiceId`와 조건 프로파일의 `providerGrantServiceId`가 중복될 수 있고, 업체·콘텐츠·조건·프로필 버전을 나타내는 용어도 여러 방식으로 사용된다. 본문은 조건 프로파일 버전 변경을 다루지만 예시 JSON에는 별도의 조건 프로파일 버전 필드가 명확하지 않다.

`[평가]` 값이 다를 때 우선순위, 상속·재정의 규칙 및 검증 실패 조건을 정해야 한다. `contentRegistrationVersion`이 조건 프로파일 변경까지 포괄하는지 포함해 동일 개념에는 동일한 이름과 버전 규칙을 사용해야 한다.

### 8.8 예시 식별자의 재사용

`[기재]` 서로 다른 광고 및 결제 예시에서 동일한 결과 식별자 또는 정산 이벤트 식별자가 반복되는 부분이 있다.

`[평가]` 단순 문서 오류라 하더라도 데이터관계를 오해하게 만든다. 예시별로 고유한 값을 사용하거나, 의도적인 공통값이면 그 이유를 설명해야 한다.

### 8.9 외부 제공자 내부 상태에 대한 과도한 단정

`[기재]` 조건 충족 전 실제 재생정보가 “존재하지 않는다”는 취지의 표현이 있다.

`[평가]` 플랫폼이 아직 취득하지 않았다는 사실과 외부 제공자 내부에 정보가 존재하지 않는다는 사실은 다르다. 외부 시스템의 내부 생성시점을 통제하지 않는다면 플랫폼 관점의 미취득·미제공으로 한정해야 한다.

### 8.10 별도 자격정보 입력이 필요 없는 근거

`[기재]` 제공자 전용 앱이나 제공자 로그인을 줄이는 효과가 강조되지만, `vendorLoginRequired=false`와 같은 설정값 외에 그 효과를 가능하게 하는 권리 근거가 충분히 설명되지 않는다.

`[평가]` 플랫폼과 제공자 사이의 계약상 권한, 플랫폼 이용자 식별을 제공자가 신뢰하는 방식, 제공자 계정 자격정보를 저장·공동사용하지 않는다는 사실을 실제 구조에 맞춰 설명해야 한다. 단순 플래그는 기술적 수단이 아니다.

### 8.11 플랫폼과 단말의 최종 조건 확인 주체

`[기재]` 설명에서는 플랫폼이 조건 실행 결과를 확인해 외부 제공자 요청을 시작하지만, 단말 청구항은 단말의 확인이 권한 요청을 직접 촉발하는 것처럼 읽힌다.

`[평가]` 최신 피드백에 따라 단말은 조건을 실행하고 원천 결과를 플랫폼에 보고하는 주체로 정리하며, 플랫폼이 콘텐츠별 조건 프로파일을 적용해 충족 여부를 최종 판정하는 것으로 통일해야 한다. 플랫폼이 단말 보고만 사용하는지 광고·결제·프로모션 시스템의 서버 결과를 함께 조회하는지는 조건 유형별 실시예에서 정해야 한다.

### 8.12 조건 관련 데이터 객체의 관계

`[기재]` `conditionFulfillmentRecord`, `conditionFulfillmentProof`, `proofPayload`와 `settlementEvent`가 서로 다른 위치에서 사용된다. 콘텐츠별 조건 프로파일에는 정산 참조도 있으나, 이 객체들이 조건별 결과를 어떻게 표현해 업체에 전달하는지는 명확하지 않다.

`[평가]` 최소한 다음 관계를 구분해야 한다.

1. **조건 실행 원천 결과**: 단말·광고·결제·프로모션 시스템이 보고한 사실
2. **플랫폼 조건 충족 판정**: 원천 결과를 콘텐츠별 조건 프로파일에 적용한 판정
3. **조건 결과 데이터**: 판정과 조건 결과 프로파일에 따라 생성되어 업체에 전달되는 공통정보와 조건별 결과
4. **업체 수락 및 권한 응답**: 업체가 결과 데이터와 고유 권한정책을 확인한 결과와 발급한 권한
5. **정산 이벤트**: 조건 결과, 업체 수락, 권한 및 실제 재생 결과를 연결한 기록

현재 `proofPayload`를 구조화된 조건 결과 데이터로 구체화하거나 그 참조값을 사용하되, 업체가 권한 발급에 필요한 실제 결과를 복원·확인할 수 있어야 한다. `settlementEventId`만 보내고 모든 의미가 플랫폼 내부에 남는 구조로는 최신 피드백을 충족하기 어렵다.

### 8.13 외부 제공자의 조건정보 확인 절차

`[기재]` 모듈 설명은 외부 VOD 제공자가 조건 관련 정보를 확인해 권한 발급 여부를 판단한다고 설명하지만, 주요 흐름은 외부 제공자가 곧바로 호환 재생방식을 선택하는 것처럼 읽힌다.

`[평가]` 외부 제공자의 순서는 조건 결과 데이터 수신 → 콘텐츠별로 등록된 결과 규칙과 대상·버전 확인 → 업체 고유 권한정책 확인 → 결과 수락 또는 거절 → 수락 시 정산처리 상태 기록 → 한시적 권한과 검증된 재생구성에 대응하는 동적 재생정보 발급으로 정리해야 한다. 업체가 플랫폼의 조건 충족 판정을 처음부터 다시 수행하는 것은 아니며, 수신 결과 데이터가 업체가 사전에 수락한 규칙과 자신의 권한정책에 맞는지를 판단하는 것이다.

### 8.14 카탈로그 저장영역과 단기 저장영역의 경계

`[기재]` 상시 카탈로그에는 실제 재생정보를 저장하지 않는다고 하면서, 저장소의 입출력 설명에는 실행 시점 재생정보를 포함하는 단기 데이터가 함께 나타난다.

`[평가]` 같은 논리 저장소의 다른 영역인지 완전히 다른 저장소인지, 상시 카탈로그와 메모리·단기 저장영역 사이의 접근권한과 삭제시점을 구분해야 한다. 그렇지 않으면 부정형 저장 한정의 의미가 흔들린다.

### 8.15 로그 정책

`[기재]` 실제 URL을 로그에서 제외한다는 설명과, 로그에 남는 경우 마스킹 또는 암호화한다는 설명이 함께 있다.

`[평가]` 평문 기록을 금지하되 보호된 형태의 기록은 허용하는 것인지, URL 전체를 기록하지 않고 식별자만 남기는 것인지 하나의 규칙으로 통일해야 한다.

### 8.16 등록 검증과 실행 시 재생구성의 충돌

`[기재]` 등록 단계에서는 정적 재생 연동 프로필에 따라 실제 재생 가능성을 시험하지만, 실행 단계에서는 외부 VOD 제공자가 정적 프로필과 단말 능력의 공통범위에서 재생방식을 새로 선택한다고 한다.

`[평가]` 최신 피드백과 실제 구현을 따르면 실행 시 자유선택이 아니라 등록 시 실제 통과한 콘텐츠별 재생구성의 재사용이어야 한다. 하나의 검증구성이면 선택단계를 삭제하고, 복수이면 검증된 재생구성집합과 단말 지원범위 안에서만 고르도록 해야 한다. 업체 응답의 프로필 버전이나 구성이 검증 결과와 다르면 원인을 업체 과실로 단정하기보다 `등록 버전 불일치` 또는 `검증되지 않은 실행 구성`으로 판정하고, 동적 재생정보 전달 차단과 재검증 상태 전환을 명시해야 한다.

### 8.17 정적 재생구성과 동적 재생정보의 경계

`[평가]` 스트리밍 방식, DRM 체계, 코덱 및 필요한 보안수준처럼 등록·검증 가능한 재생구성과, 세션마다 바뀌는 매니페스트 위치, 접근 토큰, 권한 식별자 및 DRM 요청정보를 구분해야 한다. 실행 시 동적 값이 등록 시점과 다른 것은 정상일 수 있으며, 불일치 확인 대상은 프로필 식별자·버전과 검증된 재생구성이다.

## 10. 발명자와 반드시 확인해야 할 설계 사실

### 10.1 최신 피드백으로 방향이 확정된 사항

1. 사전조건은 콘텐츠별 조건 프로파일에 등록한다.
2. 단말은 조건 실행과 원천 결과 보고를 담당하고, 플랫폼이 콘텐츠별 사전조건 충족 여부를 최종 판정한다.
3. 플랫폼은 조건이 충족되면 그 조건에 대응하는 결과 데이터를 생성해 외부 VOD 제공자에 제공한다.
4. 외부 VOD 제공자는 결과 데이터의 적합성과 업체 고유 권한정책을 확인한 뒤 원 콘텐츠 재생 권한을 발급한다. 광고 정산 또는 플랫폼 부담 프로모션처럼 정산이 발생하는 유형은 결과 데이터를 사후 정산 근거로 보관하고, 업체 부담 무료 프로모션처럼 별도 플랫폼 콘텐츠 비용이 없는 유형은 정산 없음 상태로 처리한다.
5. 콘텐츠 등록 시 실제 재생 가능성을 검증하며, 실행 시점에는 등록 단계에서 검증된 재생구성만 허용한다.
6. 실행 시점 응답이 등록·검증 정보와 다르면 정상 구성으로 받아들이지 않고 등록정보 불일치 또는 재검증 필요 상태로 처리한다.

### 10.2 여전히 설계자료로 확정해야 할 사항

각 답은 “필수구성”, “선택 실시예”, “문서에 없음”, “추가 확인 필요” 중 하나로 표시해야 한다. 문서에 없는 답을 확인 없이 발명 구성으로 가져오면 안 된다.

1. 광고 결과 데이터에 광고·캠페인 식별정보, 완료시각, 적용 배분규칙, 계산 결과 중 무엇을 필수로 포함하는가?
2. 업체 부담 무료 프로모션과 플랫폼 부담 프로모션 각각에서 비용부담, 업체 처리행위 및 후정산 근거를 어떤 필드와 상태로 표현하는가?
3. 외부 VOD 제공자는 조건 결과 데이터의 어떤 항목을 어떤 등록 규칙과 대조하고, 무엇을 저장하며, 어떤 경우 거절하는가?
4. 플랫폼이 최종 판정할 때 단말 보고만 사용하는가, 광고·결제·프로모션 시스템의 서버 결과를 조회하거나 함께 대조하는가?
5. 콘텐츠별 검증 재생구성은 하나인가 복수인가? 복수이면 플랫폼과 업체 중 실제로 누가 단말 호환 구성을 선택하는가?
6. 등록 시 시험은 단순 응답 확인인가, 실제 매니페스트 취득·DRM 라이선스·영상 디코딩까지 포함하는가?
7. 업체 응답의 프로필 버전 또는 재생구성이 검증 결과와 다를 때 권한 요청 거절, 추천 노출 중단, 재검증 요청 중 어떤 상태전이를 수행하는가?
8. 정적 재생구성에 포함되는 값과 세션별 동적 재생정보로 취급하는 값의 경계는 무엇인가?
9. 플랫폼은 외부 VOD 제공자의 사용자 계정, 비밀번호, 장기 접근자격 또는 공용 계정을 저장하거나 이용자 사이에 배정하는가?
10. 플랫폼은 원본 영상 파일이나 암호화된 세그먼트를 수집·복제·저장하는가, 아니면 업체 콘텐츠 식별자만 보유하는가?
11. 제공자 전용 앱 없이 재생되는 실제 환경은 웹 브라우저, WebView, 플랫폼 네이티브 플레이어 중 무엇이며 지원 DRM은 무엇인가?
12. 국가·지역·연령·단말 보안등급·동시시청 제한을 최종 집행하는 주체는 누구인가?

`[확인필요]` 답이 “아직 정하지 않았다”인 항목은 가능한 모든 선택지를 발명으로 적기보다, 발명자가 구체적으로 구성한 구조를 먼저 확정한 뒤 그 구조에 필요한 변형만 기재해야 한다.

- 외부 VOD 제공자 연동의 등록·검증·재검증 수명주기
- 검증된 콘텐츠만 추천 가능 상태로 전환하는 제어
- 콘텐츠별 조건과 결과 데이터 규격 및 업체 처리행위의 결합
- 업체의 조건 결과 수락·권한 발급·정산상태 기록을 연결하는 제공자 측 처리
- 등록 시 통과 재생구성·단말 지원범위·외부 제공자 응답의 일치 확인과 불일치 차단
- 플랫폼 방법, 외부 제공자 측 방법, 단말 재생 방법의 주체별 권리화

## 12. 최종 의견

`[평가]` 최신 피드백은 타당하며, 현재 직무발명서의 약점을 정확히 짚는다. 현재 문서는 콘텐츠별 사전조건, 플랫폼의 조건 확인, 정산 참조, 외부 VOD 제공자의 권한 발급 및 등록 시 재생검증을 이미 포함한다. 그러나 사전조건의 충족이 만드는 **업체 제공용 결과**, 업체가 그 결과를 확인·기록하는 방식, 그 확인과 권한 발급의 관계 및 등록 시 실제 통과 재생구성의 실행 시 구속력이 빠져 있다.

`[제안]` 가장 설득력 있는 진화 중심은 다음과 같다.

> 콘텐츠별 조건 프로파일에 조건 충족 시 생성할 결과 유형·데이터 규격·비용부담·업체 확인 규칙·업체 처리행위를 결합하고, 플랫폼이 조건 실행의 원천 결과를 이용해 사전조건 충족을 최종 판정한 뒤 해당 조건 결과 데이터를 외부 VOD 제공자에 전송하며, 제공자가 그 데이터를 등록 규칙과 고유 권한정책에 따라 확인해 수락·정산상태를 기록하고 원 콘텐츠 권한을 발급하는 구조. 이와 함께 콘텐츠 등록 단계에서 실제 통과한 재생구성과 프로필 버전을 저장하고, 실행 시 응답이 검증된 재생구성집합과 단말 지원범위에 속하는 경우에만 동적 재생정보를 재생기에 전달하며, 불일치하면 차단·상태 변경·재검증하는 구조.

광고의 경우 광고·캠페인 식별정보, 완료시각 및 적용 정산규칙을, 업체 부담 무료 프로모션의 경우 별도 플랫폼 콘텐츠 비용이 없다는 결과와 업체 캠페인 적용정보를, 플랫폼 부담 프로모션의 경우 업체에 대한 후정산 근거를, 사용자 결제의 경우 승인·금액·정산규칙을 조건 결과 데이터의 실시형태로 둘 수 있다. 구체 금액이나 배분비율은 가변적인 사업값이므로 독립항의 중심보다 결과 규격과 처리관계를 설명하는 종속항·실시예에 두는 것이 적절하다.

실행 시 재생방식은 현재 문서처럼 업체가 넓은 등록범위 안에서 새로 정하는 것으로 쓰지 않는 것이 맞다. 하나의 구성만 등록·검증됐다면 실행 시 선택단계가 필요 없고, 복수라면 실제 구현 주체가 검증집합과 단말 지원범위 안에서만 하나를 고른다. 업체는 선택된 검증구성에 맞는 권한과 세션별 URL·토큰·DRM 요청정보를 발급한다. 검증집합 밖 구성이나 버전 불일치는 업체 과실이라는 주관적 표현보다 관측 가능한 등록정보 불일치로 처리하고 재생을 차단해야 한다.

선행기술 관점에서는 US7711647B2가 플랫폼 측 사업규칙 처리 후 외부 DRM 시스템의 데이터 확인·권리 발급과 사전 등록 프로필·단말정보 활용을 상당 부분 함께 보여주므로 가장 큰 위험이다. 광고·결제·스폰서 부담과 콘텐츠 접근·정산의 연결도 다수 문헌에서 알려져 있다. 따라서 네 가지 조건을 열거하거나 정산비율을 보내는 것만으로는 부족하다. **조건과 결과 규격의 콘텐츠별 버전 결합, 업체 수락·정산상태·권한 발급의 연결, 등록 시 실제 통과 구성에 대한 실행 시 불일치 차단**이 발명자의 구체 설계에서 하나의 필수 연쇄인지가 진보성의 관건이다.

따라서 현재 상태의 결론은 **“피드백 방향은 채택할 가치가 높고 출원 검토도 계속할 수 있으나, 위 진화 구조를 발명자 설계로 확정하고 기준일 및 핵심 선행권리를 재대조한 뒤 청구항을 전면 재작성해야 한다”**이다. 현 직무발명서의 청구항을 그대로 제출하는 것은 권하지 않는다.
