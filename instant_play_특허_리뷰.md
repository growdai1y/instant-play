# Instant Play 직무발명서 특허 리뷰

- 검토 대상: [`instant_play_직무발명서.md`](instant_play_직무발명서.md)
- 검토일: 2026-07-21
- 검토 범위: 청구항 구조, 선행기술 대비 차별점, 기재요건·명확성, 실시가능성 및 권리화 전략에 관한 예비 검토

## 총평

이 문서는 직무발명서로서는 완성도가 높다. 데이터 구조, API 예시, 실패 흐름, DRM 경계, 청구항 후보까지 갖추어 실시가능성 기재는 비교적 충실하다. 특히 `조건 프로파일 → 조건 충족 증빙 → 업체 권한 → 실제 재생 정보`의 구분이 명료하다.

다만 현재 상태로 바로 출원하면 다음 위험이 크다.

| 항목 | 평가 | 판단 |
|---|---:|---|
| 실시가능성·기재 충실도 | 높음 | JSON, 순서도, 실패 처리까지 충분 |
| 신규성 | 중간 | 전체 조합은 추가 조사 필요 |
| 진보성 | 중간 이하 | 기존 통합 VOD, entitlement token, token exchange의 결합으로 공격받을 가능성 |
| 청구항 권리범위 | 낮음 | 독립항에 요소가 너무 많이 누적됨 |
| 침해 입증·집행성 | 중간 이하 | “저장하지 않음” 및 복수 사업자 행위가 핵심 구성으로 포함됨 |
| 출원 준비도 | 보완 후 출원 권고 | 핵심 프로토콜 중심으로 한 차례 재작성 필요 |

핵심 결론은 다음과 같다.

> 특허의 중심을 “앱 없이 재생”이나 “광고·결제 후 재생”에 두면 약하다. 가장 방어 가능한 중심은 “플랫폼이 생성한 조건 증빙을 외부 VOD 업체의 권한으로 안전하게 교환하고, 그 응답으로 세션 한정 재생 패키지를 처음 취득하는 교차 도메인 권한 교환 프로토콜”이다.

## 최우선 개선사항

### 1. 선행기술 평가의 재정렬

문서가 인용한 EP2647215A2는 현재 표에서 설명한 것보다 훨씬 가까운 선행기술이다.

이 문헌은 2010년 우선일을 가지며 다음을 이미 개시한다.

- 복수 콘텐츠 제공자의 메타데이터 통합
- 통합 EPG 및 범용/default player
- 사용자의 결제 또는 구독 후 provider에 콘텐츠 요청
- 제공자 웹사이트로 이동하지 않는 single-click 재생
- 제공자별 플레이어가 없을 때 플랫폼 기본 플레이어 사용

특히 “single click”, “provider 웹사이트로 이동하지 않는 universal space”, “결제 후 provider 요청”이 명시되어 있다. [EP2647215A2](https://patents.google.com/patent/EP2647215A2/en)

따라서 다음 요소는 독립적으로는 강한 차별점이 되기 어렵다.

- 복수 VOD 업체 메타데이터 통합
- 추천 플랫폼의 범용 플레이어
- 업체 앱·웹으로 이동하지 않는 재생
- 결제 후 콘텐츠 제공자에 요청
- 업체 콘텐츠는 업체가 보유하고 플랫폼은 메타데이터만 저장
- 콘텐츠별 식별자와 제공자 매핑

추가로 반드시 조사할 문헌과 표준은 다음과 같다.

- [US10602094B1](https://patents.google.com/patent/US10602094B1/en): 짧은 TTL의 entitlement token을 재생 요청에 첨부하고 VOD 서비스가 검증한 뒤 재생하는 구조
- [US20130283033A1](https://patents.google.com/patent/US20130283033A1/en): 단말·세션에 결합된 토큰을 검증한 후 스트리밍 복호화 정보를 제공하는 구조
- [RFC 8693 OAuth Token Exchange](https://www.rfc-editor.org/rfc/rfc8693.html): 한 종류의 증빙·토큰을 authorization server에 제출하고 목적 서비스용 토큰으로 교환하는 표준 구조
- [W3C Encrypted Media Extensions](https://www.w3.org/TR/encrypted-media-2/): 플레이어와 CDM의 DRM 라이선스 취득 및 콘텐츠 키 처리 경계

이들을 조합하면 현재 청구항 1 상당 부분에 대한 진보성 공격 논리가 만들어질 수 있다.

### 2. 청구항 1의 과도한 구성 누적 해소

청구항 1은 약 1,750자로, 다음을 모두 필수화하고 있다.

- 업체 단위 프로필 등록
- 콘텐츠 단위 별도 등록
- 추천 메타데이터
- 실제 재생 정보의 비저장
- 조건 프로파일 선택
- 조건 실행 및 증빙 생성
- 단말 능력 전송
- 업체의 호환 방식 선택
- 플랫폼의 선택 방식 재검증
- 단말 재생기에 전달

이는 단일 선행문헌을 피하는 데는 유리할 수 있지만, 경쟁사가 다음 중 하나만 다르게 구현해도 회피될 수 있다.

- 업체 프로필을 별도로 등록하지 않고 API discovery로 조회
- 플랫폼이 아닌 업체가 재생 방식을 선택
- 메타데이터 추천 없이 검색 결과에서 재생
- 실제 URL을 암호화 캐시에 잠시 저장
- 단말 능력 전체가 아닌 DRM capability 하나만 전달

가장 넓은 독립항에서는 다음 런타임 순서만 남기는 방안을 권한다.

1. 외부 업체 콘텐츠와 조건 증빙 요건의 매핑 유지
2. 조건 완료 결과를 검증하여 콘텐츠·세션·업체에 결합된 증빙 생성
3. 증빙을 외부 업체의 권한 서버에 제출
4. 업체가 증빙을 검증한 이후 발급한 세션 한정 재생 패키지 취득
5. 그 패키지를 플랫폼 계열 범용 재생기에 제공

업체 등록 분리, 추천 메타데이터, 단말 능력 협상, 시험 재생은 각각 종속항 또는 별도 독립항 후보로 내리는 편이 좋다.

### 3. “저장하지 않는다”는 부정적 한정의 축소

문서에는 비저장·비포함에 관한 표현이 약 40회 등장한다. 부정적 한정을 핵심 차별점으로 삼으면 다음 문제가 발생한다.

- 명확성·뒷받침 공격을 받기 쉽다.
- 경쟁사의 서버 내부 저장 여부를 침해소송에서 입증하기 어렵다.
- 암호화 캐시, 로그, CDN 캐시, 장애 복구 저장이 존재하면 비침해 주장이 가능하다.
- 현대 스트리밍 서비스에서 재생 시점에 단기 manifest/token을 받는 것은 흔하다.

다음과 같은 적극적 정보 흐름으로 바꾸는 편이 좋다.

> 카탈로그 레코드에는 업체 콘텐츠 식별자와 권한 서비스 참조 값을 유지하고, 조건 증빙이 검증된 권한 요청에 대한 응답으로 세션별 재생 패키지를 취득하며, 해당 패키지를 세션 저장 영역에 유효기간 동안 유지한다.

절대적인 “저장 금지”보다 다음 관계를 청구해야 한다.

- 조건 검증 전에는 세션 재생 패키지가 생성·취득되지 않음
- 취득한 패키지가 특정 증빙·세션·콘텐츠에 결합됨
- 카탈로그 값만으로는 직접 재생할 수 없음
- 세션 패키지는 다른 세션에서 유효하지 않음

### 4. 조건 증빙 보안 프로토콜의 구체화

현재 가장 특허성이 기대되는 부분이지만 핵심 보안 필드가 선택사항으로 기재되어 있다.

- `platformSignature`: 선택
- `vendorSignature`: 선택
- `proofSignature`: 선택

서명 없는 클라이언트 광고 재생 결과만으로 플랫폼이 증빙을 만들면 위조 방지 효과가 약하다. 적어도 하나의 강한 실시예에는 다음을 넣어야 한다.

- 증빙 발급자 또는 원천 검증자 식별자
- 업체 권한 서버를 지정하는 `audience`
- 업체, 콘텐츠, 세션, 조건 프로파일, 권한 범위
- 조건 실행 결과 또는 원천 증빙의 digest
- 발급·유효 시각
- nonce 또는 `jti`
- 서명 및 검증 키 식별자
- 증빙 사용 상태의 원자적 처리
- 동일 증빙을 다른 요청에 사용하면 거절하는 규칙
- 동일 요청 식별자의 재전송은 같은 결과를 반환하는 idempotency 규칙

RFC 8693도 일반적인 token exchange 자체는 입력 토큰을 자동으로 단회 소모시키지 않는다고 명시한다. 따라서 본 발명은 단순 token exchange가 아니라 “콘텐츠 재생용 증빙의 원자적 소모와 세션 패키지 결합”까지 내려가야 차별성이 생긴다.

### 5. 분산 실시 문제를 피하는 주체별 독립항

현재 시스템 청구항은 플랫폼, 사용자 단말, VOD 업체 시스템의 동작이 혼합되어 있다. 실제 상용 환경에서는 서로 다른 법인이 각 구성요소를 운영하므로 직접침해 입증이 어려워질 수 있다.

권장 독립항군은 다음과 같다.

- 플랫폼 서버가 수행하는 방법
- 외부 VOD 업체의 권한 서버가 수행하는 방법
- 플랫폼 서버 장치
- 업체 권한 서버 장치
- 사용자 단말 또는 스마트 TV가 수행하는 방법
- 프로그램 또는 컴퓨터 판독 가능한 기록매체

특히 현재는 “VOD 업체가 조건 증빙을 검증하고 세션 재생 패키지를 발급하는 방법”에 관한 업체 측 독립항이 없다. 이것이 중요한 누락이다.

## 구체적인 기재상 모순과 불명확성

### 검증 결과의 생성 주체

콘텐츠 등록 JSON에 `verification.status=passed`가 포함되어 있어 업체가 검증 성공을 제출하는 것처럼 보인다. 반면 동작 흐름에서는 플랫폼의 VOD 재생 검증부가 시험 재생 후 검증 상태를 생성한다.

다음 두 레코드로 분리하는 것이 좋다.

- 업체가 제출하는 콘텐츠 등록 레코드
- 플랫폼이 생성하고 서명하는 검증 결과 레코드

검증 결과에는 콘텐츠 등록 버전, 업체 프로필 버전, 시험 단말 프로필, 시험 시각 및 유효기간을 결합해야 한다.

### `vendorLoginRequired=false`

이 값은 현재 즉시 재생 콘텐츠에서 반드시 거짓이어야 하는 정책 플래그이다. 그러나 플래그만으로 업체 로그인이 기술적으로 제거되지는 않는다.

긍정적인 처리로 표현해야 한다.

- 권한 요청이 업체 사용자 계정 credential을 포함하지 않음
- 업체는 사용자 계정 인증 대신 플랫폼 서명과 조건 증빙을 검증함
- 필요한 경우 사용자 대신 콘텐츠·세션에만 결합된 가명 식별자를 사용함

또한 “완전 무계정 방식”과 “플랫폼 가명 또는 연합 인증을 이용하되 업체 앱 로그인이 없는 방식”을 별도 실시예로 두면 회피 가능성을 줄일 수 있다.

### 단말 결합 값

현재 `deviceBindingHash` 예시는 단말 capability의 해시이다. 스트리밍·DRM·코덱 목록은 동일 모델 간에 같을 수 있어 단말 고유 결합을 증명하지 못한다.

다음 변형을 추가하는 것이 좋다.

- 단말 보안영역의 공개키 또는 그 digest
- CDM이 제공한 익명·가명 단말 식별값
- 원격 attestation 결과
- 세션 공개키
- 단순 capability binding과 강한 device binding의 구별

### 시간 범위

예시에서는 900초 재생구간과 120초 권한 TTL이 함께 사용된다. 다음을 명확히 구분해야 한다.

- 증빙을 권한으로 교환할 수 있는 시간
- manifest/token을 최초 사용할 수 있는 시간
- 재생이 허용되는 콘텐츠 구간
- 시작된 재생 세션이 계속 유효한 시간

그렇지 않으면 120초 후 900초 구간 재생이 중단되는지 불명확하다.

### 네트워크 위치 정보

추천 메타데이터에는 `posterImageUrl`을 저장하면서 “네트워크 위치 정보를 저장하지 않는다”고 표현한다. 청구항에서는 다음처럼 용어를 분리하는 편이 안전하다.

- 표시용 자산 위치 정보
- VOD 미디어 전달 위치 정보
- DRM 라이선스 서비스 위치 정보
- 권한 서비스 endpoint

### 청구항 문언

- 청구항 1의 “조건 프로파일은 … 포함하는 단계”는 프로파일 정의와 방법 단계가 혼합된 문장이다.
- 청구항 16은 “검증된 콘텐츠”를 추천한다고 기재하지만 독립항 안에 검증부나 검증 기준이 없다.
- 청구항 18은 단말 조건 실행 확인부가 조건 충족을 확인하면 권한 취득을 요청한다고 기재하지만, 다른 실시예에서는 플랫폼 조건 처리부가 증빙을 생성한다. 단말의 확인과 서버의 신뢰 가능한 검증을 구별해야 한다.
- 청구항 13의 정산은 사업 규칙 성격이 강하므로 후순위 종속항으로 유지하는 것이 적절하다.
- 청구항 19의 기록매체 형식 자체는 한국 실무상 사용 가능한 형식이다. 다만 프로그램 청구항도 관할국별로 함께 검토할 수 있다. ## 권장 청구항 포트폴리오

### 핵심 출원

가장 우선할 발명은 다음 조합이다.

> 신뢰 가능한 조건 실행 결과 → 콘텐츠·세션·업체에 결합된 조건 증빙 → 외부 업체의 검증 및 증빙 소모 → 업체 발급 세션 재생 패키지 → 플랫폼 범용 재생기

세션 재생 패키지는 다음 중 하나 이상을 포함하도록 넓게 정의하는 것이 좋다.

- 단기 manifest locator
- 콘텐츠 접근 credential
- DRM license 요청 credential
- 이들을 취득하기 위한 일회성 handle
- 업체가 서명한 재생 구성 정보

실제 URL을 응답 본문에 직접 포함하지 않고 별도 endpoint에서 가져오는 구현도 포섭해야 한다.

### 별도 독립항 또는 분할 후보

1. **버전 결합 등록 검증**

   업체 프로필·콘텐츠 버전에 결합된 시험 권한 및 시험 재생 결과를 만들고, 버전 변경 시 자동으로 추천을 비활성화하는 구조이다.

2. **조건 증빙의 원자적 소모**

   동일 증빙의 중복 사용과 네트워크 재시도를 구별하는 idempotent 권한 교환 구조이다.

3. **프로필 협상 및 검증**

   업체 프로필, 콘텐츠 검증 결과, 실제 단말 능력의 교집합으로 방식을 선택하고 그 선택을 서명된 응답으로 증명하는 구조이다.

4. **구간별 권한 갱신**

   추가 조건 완료에 따라 다음 구간에만 유효한 재생 패키지를 발급하는 구조이다.

현재 문서의 Variation A~L을 모두 동등한 분할 후보로 보기보다는 위 네 묶음으로 재편하는 편이 일관성이 높다.

## 출원 명세서에서 분리할 내용

현재 문서는 직무발명서이므로 내부 검토 내용이 포함되어도 괜찮다. 하지만 정식 출원 명세서에는 다음을 그대로 옮기지 않는 것이 좋다.

- “회피 및 차별 포인트” 표
- “기존 DRM 자체는 신규하지 않다” 같은 자인성 표현
- “광고·토큰·DRM은 각각 알려져 있다”는 포괄적 선행기술 자인
- 등록 가능성 자체 평가
- 분할출원 전략
- 청구항 작성 전략
- “완료/보강 완료” 체크리스트

이 내용은 변리사 검토 메모로 별도 보관하고, 출원 명세서의 배경기술은 필요한 범위에서 중립적으로 작성해야 한다.

## 출원 전 반드시 확보할 사실자료

- 최초 발명일과 외부 공개·제안·데모 이력
- 실제 발명자별 기여 내용
- 회사·발명자의 관련 선출원
- 업체 측 권한 서버가 실제로 검증할 증빙 형식
- 증빙 발급자와 검증 키의 관리 주체
- 중복 사용 방지 및 재시도 처리 방식
- 업체 계정이 전혀 없는지, 가명·연합 인증이 있는지
- 실제 manifest·토큰·DRM 요청정보의 생성 시점
- 플랫폼, 업체, 단말 중 누가 각 데이터를 생성·검증하는지
- 실제 구현에서 가장 피하기 어려운 필수 구성

한국에서는 특허법 제29조에 따라 신규성·진보성이, 제42조에 따라 실시가능성·명확성·뒷받침이 문제된다. [특허법 제29조](https://www.law.go.kr/lsLawLinkInfo.do?chrClsCd=010202&lsJoLnkSeq=1000689898), [특허법 제42조](https://www.law.go.kr/lsLawLinkInfo.do?chrClsCd=010202&lsJoLnkSeq=1000690127). BM 발명 역시 단순 사업 규칙의 컴퓨터화만으로는 부족하고 기술적 수단과 구체적으로 결합되어야 한다는 것이 공식 심사 방향이다. [지식재산처 BM 심사 안내](https://www.kipo.go.kr/ko/kpoContentView.do?menuCd=SCD0200231)

최종적으로 이 문서는 “출원 가능한 수준의 발명서”이지만, 현재 청구항 그대로의 등록 가능성과 권리 실효성은 높게 평가하기 어렵다. EP2647215A2를 주선행문헌으로 놓고 독립항을 다시 설계한 뒤 출원하는 것을 권한다.

> 이 문서는 예비 검토 결과이며, 등록 가능성·침해 여부에 관한 공식 법률의견이나 정식 선행기술조사를 대체하지 않는다.
